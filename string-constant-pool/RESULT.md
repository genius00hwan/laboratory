# String Pool GC 실험 보고서

## 1. 개요

본 문서는 `StringPoolGcExperiment.java`를 기반으로 문자열 리터럴, `new String(...)`으로 생성한 힙 객체, `intern()` 적용 객체의 생존 특성을 비교한 결과를 정리한 것이다. 실험의 목적은 다음 질문에 답하는 데 있다.

- 문자열 리터럴은 왜 일반 힙 객체와 다르게 보이는가
- `intern()`은 문자열을 영구적으로 유지하는 기능인가
- `new String(...)`으로 생성한 객체와 풀 객체는 GC 관점에서 어떻게 구분되는가

실험은 `WeakReference`와 `ReferenceQueue`를 이용하여 강한 참조 제거 이후 객체의 생존 여부를 관찰하는 방식으로 수행하였다.

## 2. 배경

### 2.1 JVM 관점에서의 관련 개념

Java 문자열의 동작을 이해하려면 다음 세 가지를 구분할 필요가 있다.

| 구분 | 역할 |
| --- | --- |
| Heap | 일반 객체 인스턴스가 저장되는 영역 |
| Runtime Constant Pool | 클래스 로딩 시 상수 정보를 런타임 형태로 보관하는 논리적 구조 |
| String Pool | 동일한 문자열 값을 대표하는 canonical `String` 객체를 재사용하기 위한 메커니즘 |

문자열 리터럴은 컴파일 시 클래스의 constant pool 항목으로 기록되며, 실행 시 해당 값에 대응하는 canonical 문자열이 사용된다. 반면 `new String(...)`은 동일한 내용을 갖더라도 별도의 힙 객체를 추가로 생성한다.

### 2.2 `intern()`의 의미

`String.intern()`의 본질은 문자열을 canonical 객체 기준으로 공유하게 만드는 데 있다.

- 동일한 값의 풀 객체가 이미 존재하면 그 참조를 반환한다.
- 존재하지 않으면 현재 문자열을 풀에 등록하거나 해당 문자열을 대표 객체로 사용한 뒤 그 참조를 반환한다.

따라서 `intern()`은 중복 문자열 제거와 동일성 공유를 위한 기능이지, 객체의 무기한 생존을 보장하는 기능으로 해석해서는 안 된다.

## 3. 실험 목적

본 실험은 다음 세 가지 가설을 검증하기 위해 설계되었다.

1. 문자열 리터럴은 일반 힙 객체와 동일한 방식으로 회수되지 않는다.
2. `new String(...)`으로 생성한 객체는 강한 참조가 제거되면 일반 객체처럼 GC 대상이 된다.
3. `intern()` 적용 여부만으로 객체의 생존이 보장되지는 않으며, 리터럴과의 연결 여부가 결과에 영향을 준다.

## 4. 실험 설계

### 4.1 실험 대상

1차 실험에서는 다음 네 가지 참조를 비교하였다.

| 구분 | 생성 방식 | 의도 |
| --- | --- | --- |
| 리터럴 문자열 | `String lit = "Literal";` | 클래스 리터럴과 연결된 문자열 확인 |
| 일반 힙 문자열 | `new String("Heap")` | 독립 힙 객체의 회수 여부 확인 |
| `intern()` 호출 전 객체 | `new String("Intern")` | 풀 등록 전 원본 객체의 회수 여부 확인 |
| `intern()` 반환 객체 | `internObj.intern()` | 풀 객체의 생존 여부 확인 |

2차 실험에서는 리터럴 효과를 배제하기 위해 런타임에만 생성되는 동적 문자열을 사용하였다.

```java
String dynamic = new StringBuilder()
    .append("dyn-")
    .append(System.nanoTime())
    .toString();

String pooled = dynamic.intern();
```

이 방식은 최종 문자열 값이 소스 코드 리터럴로 직접 등장하지 않도록 하여, 리터럴과 무관한 `intern()` 객체의 생존 특성을 분리해서 관찰하기 위한 것이다.

### 4.2 측정 방법

각 문자열은 `WeakReference<String>`로 감싸고, 모든 강한 참조를 `null`로 제거한 뒤 GC를 반복 유도하였다. 이후 `WeakReference#get()`과 `ReferenceQueue`를 통해 객체가 여전히 도달 가능한지 확인하였다.

실험 절차는 다음과 같다.

1. 문자열 생성
2. `WeakReference` 등록
3. 강한 참조 제거
4. `System.gc()` 및 메모리 압박을 통한 GC 유도
5. 각 참조의 생존 여부 확인

## 5. 실험 결과

### 5.1 1차 실험 결과

관찰 결과는 다음과 같았다.

| 대상 | GC 이후 생존 여부 | 관찰 결과 |
| --- | --- | --- |
| 리터럴 문자열 | 유지 | `WeakReference#get() != null` |
| `new String("Heap")` | 회수 | `WeakReference#get() == null` |
| `intern()` 호출 전 힙 객체 | 회수 | `WeakReference#get() == null` |
| `intern()` 반환 풀 객체 | 유지 | `WeakReference#get() != null` |

`ReferenceQueue`에는 `new String("Heap")`과 `intern()` 호출 전 원본 객체가 등록되었고, 이는 두 객체가 GC 대상이 되었음을 보여준다.

### 5.2 2차 실험 결과

동적 문자열 기반 보완 실험의 관찰 결과는 다음과 같았다.

| 대상 | GC 이후 생존 여부 | 관찰 결과 |
| --- | --- | --- |
| 동적 문자열 원본 | 회수 | `WeakReference#get() == null` |
| 동적 문자열 `intern()` 결과 | 회수 | `WeakReference#get() == null` |

추가로 `dynamic == pooled`는 `true`로 관찰되었다. 즉, 동일 값이 풀에 존재하지 않았기 때문에 동적 문자열 자신이 canonical 객체로 사용되었음을 의미한다.

## 6. 해석

### 6.1 리터럴 문자열이 유지된 이유

리터럴 문자열은 클래스의 상수 정보와 연결되어 재사용 가능한 문자열로 취급된다. 따라서 일반적인 애플리케이션 환경에서는 해당 클래스를 통해 다시 접근될 수 있는 상태가 유지되며, 실험 시점에도 객체가 살아남는 결과가 관찰되었다.

중요한 점은 이것이 "문자열 리터럴은 절대 GC되지 않는다"는 의미는 아니라는 점이다. 실제 생존 기간은 클래스 로더의 수명과 구현체의 동작 방식에 영향을 받는다. 다만 일반적인 실행 환경에서는 리터럴이 일반 힙 객체보다 훨씬 오래 유지되는 것으로 관찰된다.

### 6.2 `new String(...)` 객체가 회수된 이유

`new String(...)`은 기존 문자열 값과 동일한 내용을 갖는 별도 객체를 힙에 생성한다. 이 객체는 리터럴과 값만 같을 뿐 동일한 객체가 아니며, 강한 참조가 제거되면 일반 힙 객체와 동일하게 GC 대상이 된다.

따라서 다음 코드는 보통 두 개의 객체를 전제한다.

```java
String s = new String("hello");
```

- `"hello"`에 대응하는 canonical 풀 객체 1개
- `new String(...)`으로 생성된 힙 객체 1개

### 6.3 `intern()` 반환 객체가 1차 실험에서 유지된 이유

1차 실험의 `"Intern"`은 코드에 리터럴로 직접 등장한다. 이 경우 `intern()`이 반환한 객체는 해당 리터럴 값의 canonical 객체와 연결된다. 따라서 `intern()` 결과가 살아남은 이유를 단순히 "`intern()`을 호출했기 때문"으로 해석하는 것은 부정확하다.

보다 정확한 해석은 다음과 같다.

- `intern()`은 동일 값을 대표하는 canonical 객체를 반환한다.
- 해당 값이 리터럴과 연결되어 있으면 객체가 장기간 유지될 가능성이 높다.
- 따라서 1차 실험에서 유지된 것은 "`intern()` 자체의 영속성"이 아니라 "리터럴과 연결된 canonical 객체의 생존"으로 보는 편이 정확하다.

### 6.4 동적 문자열 `intern()` 결과가 회수된 이유

2차 실험은 이 지점을 분리하여 보여준다. 동적으로 생성된 문자열 값은 클래스 리터럴과 직접 연결되지 않으므로, `intern()` 이후에도 별도의 강한 참조가 존재하지 않으면 GC 대상이 될 수 있다.

이 결과는 다음 사실을 시사한다.

- `intern()`은 canonical 객체를 공유하게 만들 뿐이다.
- `intern()` 적용 객체도 더 이상 강하게 참조되지 않으면 회수될 수 있다.
- 따라서 "`intern()`된 문자열은 항상 살아남는다"는 일반화는 타당하지 않다.

## 7. 한계

본 실험은 현상을 설명하는 데 유용하지만, 다음과 같은 한계를 가진다.

1. `System.gc()`는 GC 수행을 요청할 뿐, 특정 시점의 회수를 절대적으로 보장하지 않는다.
2. 결과는 JVM 구현체, 버전, 옵션, 클래스 로더 구조에 따라 달라질 수 있다.
3. 본 실험은 `WeakReference` 기반 관찰 실험이며, JVM 내부 자료구조 전체를 직접 증명하는 것은 아니다.

따라서 본 문서의 결론은 "특정 환경에서 관찰된 실험적 결과와 그에 부합하는 해석"으로 이해하는 것이 적절하다.

## 8. 결론

실험 결과를 종합하면 다음과 같이 정리할 수 있다.

1. 문자열 리터럴은 일반 힙 객체와 동일한 방식으로 취급되지 않으며, 보통 더 오래 유지된다.
2. `new String(...)`으로 생성한 객체는 독립적인 힙 객체이므로 강한 참조가 제거되면 GC 대상이 된다.
3. `intern()`은 문자열을 영구 보존하는 기능이 아니라 canonical 객체를 공유하게 만드는 기능이다.
4. `intern()`된 문자열의 생존 여부는 리터럴과의 연결, 클래스 로더의 생존, 구현체의 관리 방식에 의해 달라질 수 있다.

즉, 본 실험은 다음 문장을 가장 핵심적인 결론으로 제시한다.

> 리터럴 문자열이 오래 살아남는 현상과 `intern()`의 동작을 동일한 개념으로 단순화해서는 안 된다. `intern()`은 canonicalization 메커니즘이며, 객체의 생존 보장은 별개의 문제다.

## 9. 실무적 시사점

- 불필요한 `new String(...)` 사용은 동일 값을 갖는 중복 객체 생성을 유발한다.
- 문자열 비교는 객체 동일성(`==`)이 아니라 내용 비교(`equals`)를 기본 원칙으로 삼는 것이 안전하다.
- `intern()`은 반복도가 높고 값의 종류가 제한된 문자열 집합에 대해서만 신중히 적용해야 한다.
- 메모리 문제를 분석할 때는 "문자열 자체"보다 "어떤 참조 구조가 그 문자열을 붙잡고 있는가"를 우선적으로 확인해야 한다.
