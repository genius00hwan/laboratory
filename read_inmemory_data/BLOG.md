# 다 메모리에서 읽는거잖아?

MySQL에서 커버링 인덱스를 잘 잡으면 조회가 꽤 빨라진다.  

그래서 든 의문이..

> 인덱스도 결국 버퍼 풀에 올라와서 메모리에서 읽는 거라면, Redis 캐시와 뭐가 그렇게 다를까?

실제로 실무에서 캐시 도입 여부를 고민할 때 헷갈리는 지점이기도 하다고 한다.

## 1. 사전 지식

### MySQL도 메모리에서 읽기가 충분히 가능하다

MySQL InnoDB는 버퍼 풀(buffer pool)에 테이블/인덱스 페이지를 캐시한다.  
즉 자주 읽는 인덱스는 이미 메모리에 올라와 있을 수 있다.

그래서 어떤 조회는 실제 실행 시점에 디스크 접근이 거의 없을 수도 있다.

Redis가 빠른 이유를 단순히 "Redis는 메모리, MySQL은 디스크"여서 라고하면 애매한 답변인것.

### 그런데 버퍼 풀은 결과 캐시는 아니다

버퍼 풀은 조회 결과를 저장하지 않는다.  
저장하는 것은 페이지(page)다.

- 테이블 데이터 페이지
- 인덱스 B+Tree의 루트/중간/리프 페이지

즉 버퍼 풀에 올라와 있다는 말은,
재료가 메모리에 있다 정도이지 요리가 완성됐다는 뜻은 아니다.

## 실험 대상

실험에서는 아래 조회 하나만 고정해서 비교했다.

```sql
SELECT user_id, match_score
FROM user_match
WHERE university_id = 1
  AND gender = 'M'
ORDER BY match_score DESC
LIMIT 20;
```

이 쿼리를 비교 대상으로 한 이유

- 조건이 단순하다
- 정렬이 있다
- 반환 컬럼이 적다
- 상위 N개 조회라 캐시 후보가 되기 쉽다

이번 글의 핵심 비교 대상은 세 가지다.

- `plain`: 비커버링 인덱스
- `covering`: 커버링 인덱스
- `redis`: 조회 결과를 Redis에 JSON으로 캐시

## 커버링 인덱스란

먼저 비커버링 인덱스를 보자.

```sql
(university_id, gender, match_score DESC)
```

이 인덱스는 `WHERE university_id = 1 AND gender = 'M'` 조건과  
`ORDER BY match_score DESC`를 꽤 잘 처리할 수 있다.

문제는 `SELECT user_id, match_score`에서 `user_id`가 인덱스에 없다는 점이다.

InnoDB에서 보조 인덱스는 leaf node에 보조 키와 PK 값을 저장한다.  
그래서 비커버링 인덱스에서는 보조 인덱스로 후보를 찾은 뒤, 다시 PK를 이용해 클러스터드 인덱스를 따라가 실제 행을 확인해야 할 수 있다.

이 추가 단계가 흔히 말하는 table lookup 비용이다.

반면 커버링 인덱스는 이렇게 만든다.

```sql
(university_id, gender, match_score DESC, user_id)
```

여기서는 조회에 필요한 컬럼이 인덱스 안에 모두 들어 있다.  
즉 MySQL이 인덱스 leaf만 보고 결과를 만들 수 있다.

커버링 인덱스가 줄여주는 비용은..

- 클러스터드 인덱스 재조회 감소
- 추가 페이지 접근 감소
- 랜덤 lookup 감소
- `Using index` 가능

주의할 점은, 커버링 인덱스가 table lookup은 없애도, **"쿼리 처리 자체"**를 없애는 것은 아니라는 사실!

MySQL은 여전히 다음 일을 해야 한다.

- SQL 실행
- 인덱스 트리 탐색
- 조건 구간 찾기
- 리프 스캔
- 결과 조립

즉 커버링 인덱스로 조회 비용을 줄일 수 있다.

## Redis 구조 설명

이제 Redis를 보자.

이번 실험에서 Redis는 아래처럼 썼다.

- 키: `user_match:u=1:g=M:limit=20`
- 값: 조회 결과 20건 전체를 담은 JSON 문자열

즉 Redis는 이번 요청에서 SQL을 실행하지 않는다.  
이미 계산해 둔 결과를 키 하나로 바로 반환한다.

Redis 쪽 흐름은 훨씬 단순하다.

1. 키를 해시 테이블에서 찾는다
2. 값 객체를 꺼낸다
3. 그 값을 그대로 반환한다

이 구조가 중요한 이유는, Redis가 들고 있는 것은 **"완성된 요리"**라는 부분이다.

같은 "메모리"라고 해도 실제 성격은 완전히 다르다.

- InnoDB 버퍼 풀: 페이지 캐시
- 커버링 인덱스: 결과를 만들기 위한 접근 경로
- Redis 캐시: 최종 결과 자체

이 차이를 비유하면..

- 버퍼 풀: 재료 창고를 가까운 곳에 가져다 둔 상태
- 커버링 인덱스: 재료 위치를 명확히 해둔 상태
- Redis 캐시: 완제품을 냉장고에 넣어둔 상태

즉 Redis가 빠른 이유는 단순히 DRAM에 있음 뿐 아니라, 결과를 다시 계산하지 않고 바로 재사용하기 때문이다.

## 4. 실험

### 다시보는 실험 대상

```sql
SELECT user_id, match_score
FROM user_match
WHERE university_id = 1
  AND gender = 'M'
ORDER BY match_score DESC
LIMIT 20;
```

### 실험 환경

- MySQL 8.0.36
- Redis 7.2
- Docker Compose
- Python 적재/벤치 스크립트
- 데이터 2,000,000건
- 조회 조건 매칭 rows: 405,919
- 매칭 비율: 20.30%

실행 계획은 아래처럼 확인됐다.

- `plain`: `Using index condition`
- `covering`: `Using where; Using index`

즉 `plain`은 인덱스를 타지만 결과를 인덱스만으로 끝내지 못했고,  
`covering`은 인덱스만으로 결과를 만들 수 있었다.

### 시나리오

세 가지 상황으로 나눔.

| scenario | description | 전체요청수 | warmup | 동시요청수 |
|---|---|---:|---:|---:|
| interactive_single | 단건 응답 시간 위주. 사용자 한 명이 반복 조회 | 3000 | 300 | 1 |
| steady_api | 일반적인 API 핫패스 | 8000 | 500 | 8 |
| burst_hot_key | 같은 조회가 짧은 시간에 몰리는 버스트 | 16000 | 1000 | 32 |

### 결과

| scenario | case | avg_ms | p95_ms | max_ms | tps |
|---|---|---:|---:|---:|---:|
| interactive_single | plain | 0.828 | 1.546 | 29.800 | 1207.25 |
| interactive_single | covering | 0.888 | 1.536 | 13.499 | 1176.80 |
| interactive_single | redis | 0.401 | 0.650 | 9.462 | 2489.13 |
| steady_api | plain | 2.930 | 6.102 | 29.682 | 2693.88 |
| steady_api | covering | 1.853 | 3.743 | 10.998 | 4280.10 |
| steady_api | redis | 0.774 | 1.307 | 5.193 | 10277.76 |
| burst_hot_key | plain | 13.031 | 26.904 | 93.980 | 2401.61 |
| burst_hot_key | covering | 12.099 | 27.690 | 115.777 | 2618.47 |
| burst_hot_key | redis | 1.977 | 3.940 | 50.603 | 16067.44 |

### 결과 해석

#### 단건 저부하에서는 커버링 인덱스 차이가 크지 않았다

`interactive_single`에서는 `plain`과 `covering`이 거의 비슷했다.

- plain avg: `0.828ms`
- covering avg: `0.888ms`
- plain p95: `1.546ms`
- covering p95: `1.536ms`

이건 이상한 결과가 아니다.  
이번 쿼리는 `LIMIT 20`이라서, 비커버링 인덱스여도 결국 재조회하는 행 수가 작다.  
즉 저부하 단건 상황에서는 table lookup 비용 자체가 절대 시간으로 매우 작다.

#### 중간 부하부터는 커버링 인덱스 효과가 보였다

`steady_api`에서는 커버링 인덱스가 확실히 유리했다.

- avg: `2.930ms -> 1.853ms`
- p95: `6.102ms -> 3.743ms`
- TPS: `2693.88 -> 4280.10`

단건에서는 작아 보이던 table lookup 비용이, 반복 부하에서는 누적되어 실제 차이로 드러난 것이다.

#### 버스트에서는 Redis가 사실상 다른 급이었다

`burst_hot_key`에서는 커버링 인덱스가 평균만 조금 나아졌고 p95는 거의 개선되지 않았다.

- plain avg: `13.031ms`
- covering avg: `12.099ms`
- redis avg: `1.977ms`

- plain p95: `26.904ms`
- covering p95: `27.690ms`
- redis p95: `3.940ms`

여기서는 병목이 lookup을 몇 번 줄이느냐 문제를 넘어선다.  
MySQL은 여전히 SQL 엔진과 InnoDB를 타서 쿼리를 처리해야 한다.  
반면 Redis는 같은 결과를 키 하나로 바로 반환한다.

그니까 버스트 구간에서는 커버링 인덱스와 Redis의 차이가 구조의 차이로 수치의 차이를 만드는 느낌스.

## 5. 배운 점

이번 실험에서 가장 중요한 포인트는 하나다.

> "둘 다 메모리에 있다"는 말만으로는 설계 판단이 안 된다.

### 1. 커버링 인덱스는 먼저 시도해볼 만한 최적화다

다음 조건이면 우선 커버링 인덱스로 끝내는게 맞는듯

- 조회 패턴이 단순하다
- 반환 행 수가 작다
- avg와 p95가 이미 충분히 낮다
- 최신성이 중요하다
- 캐시 무효화 복잡도를 늘리고 싶지 않다

즉 DB 최적화만으로도 충분한 상황이 분명히 있음

### 2. Redis는 조회 처리를 우회하는 강점이 있음

Redis를 붙이는 순간 얻는 이점은 단순한 미세 최적화가 아니다.

- SQL 실행 우회
- 인덱스 탐색 우회
- 결과 재조립 우회
- hot key 반복 요청 흡수

그래서 같은 조회가 계속 반복되는 구간에서는 Redis가 훨씬 큰 효과를 낸다.

### 3. 절대 시간이 중요하다

이건 실무에서 특히 중요하다.

- `0.8ms -> 0.4ms` 개선
- `12ms -> 2ms` 개선

둘 다 퍼센트로는 좋아 보일 수 있지만 운영 의미는 전혀 다르다.

전자는 "좋긴 한데 굳이?"일 수 있고,  
후자는 "이건 구조적으로 바꿀 이유가 있다"에 가깝다.

### 4. p95가 무너지면 Redis 도입 명분이 커진다

평균만 보면 착각하기 쉽다.  
실제 체감은 p95, p99 같은 꼬리 지연에서 더 크게 난다.

이번 실험에서도 버스트 구간에서 Redis는 p95를 크게 낮췄다.

- covering p95: `27.690ms`
- redis p95: `3.940ms`

이 수치는 사용자 체감과 시스템 여유가 달라지는 수준이다. 괜히 `tail latency`가 중요하다고들 하는게 아니다.

### 5. 실무에서는 이렇게 하라네

1. 먼저 인덱스와 쿼리 구조를 정리한다
2. 커버링 인덱스로 줄일 수 있는 비용을 먼저 줄인다
3. 그래도 hot key 반복이나 버스트로 p95가 무너지면 Redis를 붙인다

즉 `DB 최적화 -> 캐시 검토` 순서가 기본이다.  
다만 hot key가 너무 명확하고 버스트가 강하면 처음부터 Redis를 같이 보는 게 맞다.
